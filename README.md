# video-with-alpha-decoding

## 📋 Универсальное решение для видео c прозрачным фоном (MP4/AV1 + WebGL1/WebGL2)

### 💬 Глоссарий
* __WebGL2__ - [Baseline Widely available (some parts of this feature may have varying levels of support).](https://developer.mozilla.org/en-US/docs/Web/API/WebGL2RenderingContext)
* __WebGPU__ - [Is not Baseline feature because it does not work in some of the most widely-used browsers.](https://developer.mozilla.org/en-US/docs/Web/API/WebGPU_API)
* [__AV1__](https://aomedia.org/specifications/av1/) - Is an open-source, royalty-free video coding format, offering ~30% better compression than VP9/HEVC.
* [__MP4__](https://ossrs.net/lts/zh-cn/assets/files/ISO_IEC_14496-14-MP4-2020-379935634d95ce9ab9c0eca6d47c6b79.pdf) - file format (multimedia container) for store video/audio/image/subtitles/etc. data.

### 🚫 Суть проблемы:
Основными ресурсами приложения являются *тяжёлые* анимации с альфа-каналом. Стандартные подходы (WebM, HEVC, PNG-секвенции) либо не кроссбраузерны, либо избыточны по весу и потреблению памяти (🧨 '_Out Of Memory_' на iOS).

### 🎯 Задача:
Использовать один базовый формат (__H.264(AVC)__ / __AV1__ / __VP8__ внутри **MP4**) с реализацией прозрачности, минимизировать нагрузку на __CPU__ / __DOM__ и интегрировать решение в используемый _web-2D-engine_.

---

## 🛠 Предлагаемое решение

### 📝 Концепция «Alpha Packing»
Для поддержки прозрачности в стандартном __H.264__ / __AV1__ / __VP8__ используется метод **Side-by-Side** (или **Top-Bottom**) упаковки:
- Создается один видео-ассет с увеличенной в 2 раза шириной/высотой.
- В одной части кадра — RGB данные (цвет).
- В другой части — Greyscale маска (яркостная составляющая альфа-канала).

### ⚙️ Технологический стек
Для обхода ограничений DOM и для прямой передачи кадров в GPU используется комбинация:
1.  **FFmpeg**: Этап предварительная подготовки видео-ассетов, встроенный в общий ресурс-пайплайн.
2.  **WebCodecs API (`VideoDecoder`)**: Позволяет декодировать видеопоток покадрово в фоновом потоке, получая объекты `VideoFrame`.
3.  **Wasm-Video Decoding**: Для окружения, в API которого не присутствует или не в полной мере реализованы интерфейсы _WebCodecs API_, использует декодирование видеопотока средствами CPU.
4.  **Web Workers API**: Декодирование выносится из Main Thread, предотвращая влияние на динамику анимаций интерфейса и скорость исполняемого JS-кода.
5.  **PixiJS v8 + WebGPU/WebGL**: Использование кастомного шейдера для слияния RGB и Alpha областей текстуры в один проход отрисовки.

> [!IMPORTANT]
> H.264 (AVC) требует лицензионных отчислений при программном декодировании, поэтому имеет смысл использовать `бесплатные` в этом плане кодеки.

## 🏗 Архитектура реализации

### ⚡ Ключевые моменты
*   **Оптимизация памяти:** Избегаем создания промежуточных RGBA8888 текстур двойного размера. Для загрузки данных декодированного кадра анимации (для обновления существующих текстур в GPU) используем исключительно `texSubImage2D()`-метод.
*   **Буферизация:** Инициировать асинхронное декодирование кадра N+1 перед началом отрисовки кадра N.
*   **Кэширование:** Учитывать, что для декодирования каждого N кадра анимации (кроме первого кадра и I-кадров) требуются данные N-1 кадра.
*   **Синхронизация:** Предварительное декодирование первого кадра (на этапе загрузки видео-ассета) для мгновенного отображения (исключение '_черного мигания_').
*   **Восстановление контекста:** Обработка события потери контекста WebGL с автоматической перезагрузкой последнего декодированного кадра в GPU.
*   **Утилизация:** При '_выгрузке_' ресурса видео-анимации производить освобождение памяти, занимаемой декодированными данными (например VideoFrame.close()), остановка декодера с освобождением выделенной ему памяти, закрытие потока (воркера) и т.д.
*   **Эффективность:** _WebCodecs API (`VideoDecoder`)_ не требует кастомного шейдера - можно загружать данные кадра напрямую в атлас кадров или отдельную текстуру через `texSubImage2D()`-метод. А также эффективно использует GPU для операций декодирования и преобразования полученных YUV-данных кадра в RGB-формат.
*   **Совместимость:** Декодирование через _Wasm-декодер_ потребует реализации процесса преобразования YUV-плоскости в RGB-данные в кастомном шейдере. Также процесс декодирования будет происходить ресурсами CPU.
*   **3d-party ресурсы:** Демуксинг (парсинг видео-контейнера и разбивка входных данных на закодированные чанки/кадры) потребует сторонней библиотеки в обоих случаях (и при работе через _WebCodecs API_, и при использовании _Wasm-декодера_). Реализация фолбэк подхода с _Wasm-декодером_ тоже будет базироваться на opensource-библиотеке.
*   **Лицензии:** Необходимо использовать только opensource-библиотеки с MIT-лицензией или Apache-2.0. Также важно понимать, что H.264 (AVC) требует лицензионных отчислений при программном декодировании, поэтому имеет смысл использовать бесплатные AV1 либо VP8 (качество после сжатия хуже) кодеки.

### ✅ Основное решение (Primary solution)
*   **Простой подход:** на один элемент (одну ноду) с видео-анимацией на отрисовываемой сцене - один батч и отдельный draw-call, отдельная текстура (или две: одна с цветом, вторая с прозрачностью), отдельные кастомные шейдера (со смешиванием цвета и прозрачности) и программа.
*   **Продвинутый подход:** батчинг видео-анимаций. Т.е. создаётся атлас на несколько видео-анимаций, данные каждого нового N кадра каждой видео-анимации помещаются в свой выделенный фрейм этого атласа (запись через `texSubImage()`), для каждой вершины в вершинном буфере добавляется индекс текстуры, помимо текстурных uv-координат.
При переполнении атласа кадров видео-анимаций создаётся новый атлас. 
При добавлении на сцену одной или нескольких новых нод с видео-анимацией фреймы в атласе перекомпоновываются и заново загружаются закэшированные данные кадров всех активных видео-анимаций, чьи фреймы были перемещены.
Алгоритм компоновки от большего к меньшему - сначала компонуются большие фреймы, потом более мелкие и самые маленькие в конце.
Предусмотрен поворот фреймов на 90град. для более компактной упаковки их в атласе.

### 🚧 Фолбэк (Fallback)
Если `WebCodecs API` не поддерживается (старые версии браузеров), используется **WASM-декодер** внутри Worker. Это гарантирует 100% работоспособность при сохранении логики шейдерного смешивания.

## ⏳ Оценка R&D и внедрения

| Этап                        | Описание                                                                                                                                             | Сроки                |
|:----------------------------|:-----------------------------------------------------------------------------------------------------------------------------------------------------|:---------------------|
| **R&D**                     | Исследование производительности `VideoDecoder` и WASM, написание базового шейдера для PixiJS v8, тесты (в том числе на iOS / Safari).                | **3–5 рабочих дней** |
| **Разработка**              | Создание обертки (интеграция) для PixiJS (Resource/Texture Parser), логика управления потоками (Workers), обработка циклов и пауз и т.д.             | **5–7 рабочих дней** |
| **Оптимизация**             | Тюнинг управления памятью, обработка Edge-cases (потеря контекста, скачки FPS), финализация пайплайна подготовки видео, нагрузочное тестирование.    | **3–4 рабочих дня**  |
| **Модульное тестирование**  | Покрытие юнит-тестами не менее 80% добавленного кода                                                                                                 | **1-2 рабочих дня**  |
| **Автотесты/Скриншоттесты** | Создание отдельно запускаемого сэмпла (сцены с использованием видео-анимаций) и добавление его в пул автотестов/скриншоттестов, платформенных тестов | **1-2 рабочих дня**  |

**Итого: ~3-4 недели** до стабильного Production-ready решения.
